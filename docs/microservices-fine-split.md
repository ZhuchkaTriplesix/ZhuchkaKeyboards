# Декомпозиция бэкенд-сервисов: **адекватный** уровень

Edge — **Traefik**; отдельного API Gateway-приложения нет.

Список на **103** сервиса был намеренным **потолком по границам** — в эксплуатации это почти всегда лишняя боль (сеть, согласованность, релизы, команды). Ниже — **разумное** разбиение для enterprise: **отдельный деплой только там, где граница реально окупается** (разные SLO, разные команды, разный цикл изменений, изоляция PCI/безопасности).

---

## Рекомендуемый ядро: **14 сервисов**

| # | Slug | Зона ответственности |
|---|------|----------------------|
| 1 | `auth-service` | Аутентификация и авторизация: OAuth2/OIDC, пользователи, роли/права, MFA, сессии/refresh, JWKS, аудит входов. **Одна** БД auth — без отдельного «iam-service», пока нет жёсткой политики ABAC. |
| 2 | `customer-service` | Профили B2C/B2B, адреса, согласия (ПДн); связь с `sub` из Auth. |
| 3 | `catalog-service` | Каталог + PIM в одном контуре: товары, варианты, атрибуты, медиа, SEO/read-модели, **совместимость** для клавиатур. Поиск — модуль или отдельный индекс внутри же, пока нет отдельной команды search. |
| 4 | `commerce-service` | Корзина, чекаут, промо/купоны, расчёт налогов/доставки на чекауте, создание заказа продажи. |
| 5 | `payments-service` | Эквайринг: интенты, webhooks, идемпотентность, возвраты; **PCI** и секреты провайдера изолированы здесь. |
| 6 | `oms-service` | Заказ продажи: строки, статусы жизненного цикла для клиента и цеха, связь с оплатой. |
| 7 | `fulfillment-service` | Исполнение: резерв/аллокация по складам, волны, исходящие отгрузки, интеграции перевозчиков, трек к покупателю, возвраты/RMA — **всё, что «после оплаты до доставки клиенту»**, кроме чистого количества на складе. |
| 8 | `inventory-service` | Остатки, движения, резервы в количественном учёте; источник истины по qty. |
| 9 | `wms-service` | Склады, зоны, ячейки, задачи приёмки/отбора/размещения, инвентаризации. |
| 10 | `production-service` | Производство: производственные заказы, этапы, ресурсы, ОТК/брак. |
| 11 | `custom-service` | Кастом: конфигуратор (валидация, цена черновика), BOM, согласования с клиентом, связь с OMS/production. |
| 12 | `procurement-service` | Закупки: заявки, PO, входящие поставки, трекинг, приёмка (GRN), при необходимости сверка со счётами поставщика. |
| 13 | `counterparties-service` | Контрагенты: юрлица, договоры, реквизиты; связи с закупками и B2B. |
| 14 | `notification-service` | Шаблоны, очередь, каналы (email/SMS/push) — один сервис, внутри — адаптеры по каналам как модули. |

**Итого: 14** — баланс между автономией доменов и числом деплоев/БД/релизов.

## Когда имеет смысл **дробить дальше** (не по умолчанию)

| Если… | Вынести |
|--------|---------|
| Поиск/каталог нагружает CPU/Elastic отдельно от PIM | `search-service` или отдельный read-replica + worker, не обязательно третий «сервис» |
| Промо и чекаут живут разными командами и частыми релизами | `promotion-service` отдельно от `commerce-service` |
| Финансовая сверка и проводки сложнее, чем «оплатил заказ» | `ledger-service` или интеграция с ERP, не обязательно свой микросервис сразу |
| Закупки и склад спорят за границу | оставить **procurement** отдельно от **wms** (уже так в ядре) |

---

## Чего **не** плодить без нужды

- Отдельный `oauth2-server` и `credential-service` и `session-service` — внутри **`auth-service`** модулем.
- Отдельный `coupon-service` от промо — в **`commerce-service`**.
- Десяток сервисов вокруг одного трека «заказ посылки» — объединено в **`fulfillment-service`**.
- Отдельный `login-audit` — таблица/поток в **`auth-service`** + корреляция в Loki.
