# Архитектура и домен: клавиатуры, продажи, производство, кастом

Документ описывает **полноценный enterprise-контур** для ZhuchkaKeyboards: микросервисная (или строго модульная) платформа управления предприятием — **продажи** (витрина, B2C/B2B), **производство и кастом**, **закупки и входящие поставки с отслеживанием**, **контрагенты и договорной контур**, операционные и аналитические интерфейсы. Цель — уровень **корпоративной эксплуатации**: управляемость, аудит, отказоустойчивость, соответствие требованиям по данным и доступу, а не «один сайт и скрипт на сервере».

Текущий код в репозитории (PostgreSQL + Alembic, склад `items`, **поставщики (`suppliers`)**, производство `orders`, Redis, gateway) — исходная точка миграции на этот контур.

---

## Enterprise: принципы и нефункциональные требования

Ниже — то, без чего система **не считается** enterprise-готовой, независимо от домена (клавиатуры — отраслевая специфика сверху).

### Управление доступом и идентичность (IAM)

- Реализация — **отдельный сервис Auth** (раздел 2.1): он владеет учётными записями, ролями, выдачей токенов и аудитом входов; доменные сервисы **авторизуют** запросы по токену, не реализуют логин сами.
- **Единый вход** для сотрудников (SSO по возможности: OIDC/SAML), **RBAC или ABAC**: роли по контурам (склад, закупки, производство, поддержка, финансы), разграничение до уровня действий и данных (склады, юрлица, договоры).
- Разделение **учёток клиента** (витрина) и **учёток персонала** (операционка) — разные клиенты OAuth / scopes; сервисные аккаунты для интеграций, ротация ключей.

### Аудит и неизменяемость следов

- **Журналирование действий** пользователя и сервисных вызовов: кто изменил PO, кто подтвердил приход, кто отменил резерв — с timestamp, корреляцией и до/после где уместно.
- Критичные бизнес-события хранить так, чтобы удовлетворять **разбор инцидентов** и при необходимости **регуляторные / договорные** проверки (сроки хранения логов — политика).

### Данные и соответствие (compliance-ready)

- Классификация данных (ПДн, платёжные метаданные, коммерческая тайна); **шифрование** в транзите и для чувствительных полей в покое; политики **retention** и удаления по запросу субъекта где применимо.
- Разделение сред (dev/stage/prod), секреты только из **Vault / secret manager**, запрет «секретов в репозитории».

### Надёжность и непрерывность (HA / DR)

- Целевые **SLO** по сервисам (доступность API витрины, очередь заказов, складские операции); **резервное копирование** БД (PITR где нужно), **план восстановления** (RPO/RTO) для PostgreSQL и артефактов.
- Stateless-приложения за балансировщиком; **кластеризация** брокера и БД по мере нагрузки; health/readiness для оркестратора.

### Интеграции и контракты

- Внешние системы (эквайринг, доставка, бухгалтерия/ERP, EDI с поставщиками) — **версионированные API**, идемпотентность, **dead-letter** очереди, мониторинг интеграционных ошибок.
- Внутренние границы — **контракты событий** (схемы), совместимость версий при эволюции.

### Эксплуатация и наблюдаемость

- **Метрики, логи, трассировки** (при необходимости OpenTelemetry) — обязательный минимум для production; **алерты** по SLO и по бизнес-порогам (рост ошибок оплаты, провалы резерва).
- Runbook’и в MkDocs: инциденты, откат релиза, восстановление БД.

### Масштабирование организации

- Несколько складов, юрлиц, валют, каналов продаж — закладывается в **модель контрагентов, складов и организаций**, чтобы не ломать схему при расширении.

## 0. Целевой контур развёртывания (MkDocs, Traefik, фронты, бэкенды, БД, наблюдаемость)

Ниже — **целевая архитектура** (весь перечень ролей и сервисов); внедрение идёт поэтапно, без требования поднять всё единовременно.

### 0.1 Документация — MkDocs

- **MkDocs** (или MkDocs Material) для этого репозитория: архитектура, runbook’и, схемы API, глоссарий домена.
- Сборка — **статический сайт** (CI → артефакт); в проде — отдельный контейнер/nginx или хостинг статики за Traefik по пути вроде `/docs` или поддомен `docs.*`.
- Исходники остаются в `docs/`; конфиг `mkdocs.yml` в корне репозитория — когда появится.

### 0.2 Входной трафик — Traefik

- **Traefik** как **reverse proxy / ingress**: TLS (Let’s Encrypt), маршрутизация по хостам и путям, health checks, опционально rate limit на уровне маршрутов.
- Типичная раскладка:
  - `www` / основной домен → **публичный фронт**;
  - `admin` / `app` или путь `/admin` → **операционный фронт** (защита сетью + авторизацией);
  - `/api` → **gateway/BFF** или напрямую к сервисам по префиксам;
  - **`/oauth` / `auth.*`** → **сервис Auth** (token, authorize, JWKS, интроспекция — см. раздел 2);
  - опционально отдельный хост для Grafana.

### 0.3 Два фронтенда и сайт продаж

| Фронт | Назначение | Аудитория |
|-------|------------|-----------|
| **Публичный (сайт продаж)** | Витрина, каталог, конфигуратор кастома, ЛК покупателя, оплата, статусы **исходящих** отгрузок покупателю | Клиенты (B2C, B2B — по политике аккаунтов и договоров) |
| **Операционный** | Склад, производство, заказы, **закупки и входящие поставки**, контрагенты, приёмка, поддержка, настройки каталога, отчётность | Сотрудники (роли по RBAC) |

**Сайт для продаж** — это публичный фронт за Traefik на основном домене; он не заменяет учёт закупок: закупки и трекинг **входящих** поставок живут в операционном контуре и API, на витрине при необходимости показывают только производные вещи (например, «ожидаем поступление — срок уточняется»), если продукт решит это раскрывать клиенту.

Оба фронта ходят в **одни и те же или разные BFF/API** в зависимости от политики безопасности (часто один gateway с разными политиками и scope токенов).

### 0.4 Бэкенд — сервисы

- Отдельный **сервис Auth** (аутентификация и авторизация) — см. раздел 2; остальные доменные сервисы **не проверяют пароли сами**, а валидируют токены (JWKS) или спрашивают интроспекцию у Auth.
- Набор **микросервисов** из раздела 2: заказы, каталог, склад, производство/кастом, платежи, уведомления и т.д.
- Каждый сервис с **HTTP API** (и при необходимости gRPC внутри сети), **liveness/readiness** для оркестратора и Traefik.

### 0.5 Базы данных и кэш

- **PostgreSQL** — основной персистентный слой; по стратегии *database per service* — **несколько инстансов/БД или схем** под разные bounded context’ы (как минимум отделить «витрина+заказы» от «склад+производство», если границы сервисов разные).
- **Redis** — сессии, кэш API, rate limiting, очереди/стримы (в зависимости от выбора брокера), блокировки.
- При появлении **очереди событий** (RabbitMQ / NATS / Kafka) — отдельные контейнеры; это не «БД», но отдельная инфраструктурная зависимость.

### 0.6 Наблюдаемость — Prometheus, Grafana, экспортеры

- **Prometheus** — сбор метрик со scrape-конфигов по сервисам и инфраструктуре.
- **Grafana** — дашборды (SLI/SLO, бизнес-панели по заказам — через метрики или логи).
- **Экспортеры и источники** (базовый набор):
  - приложения: эндпоинт `/metrics` (Prometheus client в каждом бэкенде);
  - **node_exporter** — хост;
  - **postgres_exporter** — БД;
  - **redis_exporter** — Redis;
  - **Traefik** — встроенные метрики;
  - при необходимости **blackbox_exporter** — проверки снаружи.
- Логи: агент к **Loki** (в репозитории уже есть задел под Loki/Promtail) — связка с Grafana для единого входа.

Итого по ролям: **MkDocs (статика), Traefik, 2 фронта, сервис Auth, N доменных бэкендов, 1+ PostgreSQL, Redis, Prometheus, Grafana, экспортеры**, плюс Loki и брокер сообщений по необходимости — это **нормальный production-стек** вокруг домена, а не «лишние» микросервисы ради микросервисов.

## 1. Как устроены похожие системы в индустрии

### 1.1 Типичный разрез

| Слой | Назначение | Примеры подходов |
|------|------------|------------------|
| **Витрина / e-commerce** | Каталог, корзина, оплата, доставка, аккаунт клиента | Headless CMS + storefront (Shopify Hydrogen, Medusa, custom Next.js), или классический монолит магазина |
| **OMS (Order Management)** | Жизненный цикл заказа после оплаты: резерв, сборка, отгрузка, возвраты | Отдельный сервис или модуль ERP |
| **WMS / Inventory** | Остатки по складам, партии, серийники, перемещения | Часто отдельный сервис с жёсткой моделью движений |
| **MES / Production** | Маршруты сборки, этапы, загрузка цеха, учёт брака | Для кастома критичен BOM и привязка к конкретному заказу |
| **PIM / Catalog** | SKU, атрибуты, совместимость (например, layout PCB ↔ case) | Отдельно от складских «штук», чтобы не плодить дубли |
| **SRM / Procurement** | Заявки на закупку, заказы поставщику (PO), приёмка, сопоставление счётов | Часто модуль ERP или отдельный сервис; связан с WMS через приход на склад |
| **Контрагенты (master data)** | Юрлица/ИП: реквизиты, договоры, роли (поставщик, перевозчик, клиент B2B) | Единый справочник с привязкой к операциям закупок и продаж |
| **IAM / Auth** | Выдача токенов, роли, политики доступа, SSO | Отдельный сервис (Keycloak-подобный контур или собственный OAuth2 AS) |

Крупные игроки часто **не** делают всё с нуля: связка **витрина + OMS + WMS** с ERP (1С, NetSuite, Odoo и т.д.). Здесь цель — **собственная платформа enterprise-класса** под домен клавиатур: конфигуратор, BOM, учёт компонентов и услуг (луб и т.д.), плюс полный контур закупок и производства — с **IAM, аудитом и интеграциями**, как у корпоративного решения, а не «коробочный магазин».

### 1.2 Что важно для клавиатурного бизнеса

- **Два типа продажи**: готовый SKU с полки vs **конфигурируемый** набор (каждая позиция — выбор из совместимых вариантов).
- **BOM (bill of materials)**: для кастома заказ = дерево компонентов + опционально труд (сборка, луб, тест).
- **Совместимость**: форм-фактор (60%, 75%, TKL), тип крепления PCB, высота кейкапов, раскладка стабов — это правила, а не только текст в карточке.
- **Склад в штуках**, но учёт может быть **по партиям** (одна партия свитчей — разные характеристики после моддинга).

## 2. Микросервисы: предлагаемые границы

Микросервисы имеют смысл, когда **границы домена** и **независимое масштабирование/релизы** перевешивают операционные затраты (сеть, распределённые транзакции, наблюдаемость).

### 2.1 Вариант декомпозиции (логический)

1. **Auth (сервис аутентификации и авторизации)** — **отдельный деплой**, отдельная БД (см. раздел 3.2). Функции: **OAuth2 / OIDC** (authorization server), выдача **access / refresh** токенов, **JWKS** для проверки JWT доменными сервисами, **client registration** (публичный фронт, операционный фронт, мобильные клиенты, сервис-сервис); **интроспекция** или локальная валидация по ключам; **RBAC** (роли, permissions, scopes), опционально **ABAC / policy engine** (OPA) при росте сложности; **MFA**, блокировки учёток, **аудит входов**; пароли, секреты клиентов и refresh — только здесь; **SSO** (SAML/OIDC) к корпоративному IdP для персонала. Gateway **не** заменяет Auth: он терминирует TLS и проксирует, доверие к субъекту — по токену, выданному Auth.
2. **Directory / Customers (профили)** — B2C-профили покупателя, согласия, адреса доставки; связь `user_id` / `subject` из Auth. Может жить в Commerce или отдельным сервисом, но **идентификатор субъекта** всегда из Auth.
3. **Catalog & PIM** — товары, варианты, атрибуты, медиа, **правила совместимости** (можно хранить как данные + движок правил).
4. **Commerce / Cart & Checkout** — корзина, промокоды, расчёт доставки, создание **sales order** (без списания со склада до подтверждения оплаты — зависит от политики).
5. **Payments** — интеграция с эквайрингом; webhooks; идемпотентность.
6. **OMS** — единый номер заказа, статусы для клиента и для цеха, связь с оплатой и отгрузкой.
7. **Inventory / WMS** — склады, остатки, резервирование под заказ, отгрузка, возвраты (у вас уже богатая модель `items`, транзакции).
8. **Production / MES** — производственные заказы, этапы, загрузка, качество (частично уже есть `production`).
9. **Custom orders / Configurator** — доменная оболочка над каталогом + складом + производством: **спецификация кастома**, BOM, согласование с клиентом, сроки.
10. **Procurement / SRM** — закупки: заявки, заказы поставщику (PO), ожидаемые поставки, **отслеживание входящих отправлений** (трек, статусы, ETA), согласование с приходом на склад (Goods Receipt).
11. **Counterparties / MDM** — контрагенты: юридические лица, договоры, платёжные условия; связь с `suppliers`, клиентами B2B и перевозчиками. До выделения в отдельный сервис — часть Procurement или OMS.
12. **Notifications** — email, Telegram, push; шаблоны; очередь.
13. **API Gateway / BFF** — маршрутизация, **передача Bearer** к доменным сервисам, rate limiting, опционально первичная проверка JWT по JWKS Auth (у вас уже есть gateway-паттерн).

Первый деплой не обязан совпадать с полным списком: **Catalog + Commerce + OMS** нередко оформляются как модульные границы внутри одного артефакта, затем выносятся в отдельные сервисы по нагрузке и частоте релизов.

### 2.2 Синхронные vs асинхронные границы

- **Синхронно**: проверка наличия при добавлении в корзину, финальная валидация конфигурации перед оплатой.
- **Асинхронно (события)**: `OrderPaid` → резерв на складе; `InventoryReserved` → разрешение на сборку; `ProductionStepCompleted` → обновление ETA для клиента.

Шина событий (Kafka, RabbitMQ, NATS) + **идемпотентные обработчики** — стандарт для связки заказа и склада без распределённых 2PC-транзакций.

## 3. Базы данных: стратегия

### 3.1 Database per service

- У каждого сервиса **своя схема/БД**, если граница сервиса стабильна. Это снимает связность миграций и даёт независимые бэкапы.
- **Не шарить одну таблицу `orders` между витриной и цехом** без чёткого владельца: иначе получится распределённый монолит в SQL.

### 3.2 Что где жить логично

| Данные | Где |
|--------|-----|
| Пользователи (учётки), роли, permissions, клиенты OAuth, refresh/sessions, аудит входов | **Auth DB** (отдельно от бизнес-данных) |
| Каталог, SEO, совместимость | Catalog DB |
| Корзина (до checkout) | Redis или сервис Commerce с TTL |
| Заказ клиента (OMS) | OMS DB |
| Остатки, движения, резервы | Inventory DB |
| BOM кастома, чертежи согласований | Custom/MES DB + файловое хранилище (S3-совместимое) |
| Платежи, провайдерские id | Payments DB (PCI: не хранить лишнее) |
| Закупки (PO, строки, статусы), ожидаемые поставки | Procurement DB (или модуль в Inventory до выноса) |
| Контрагенты, договоры, реквизиты | Counterparties DB или схема в Procurement/OMS |

### 3.3 Консистентность

- **Saga / orchestration** для цепочки: оплата → резерв → постановка в очередь сборки.
- **Outbox pattern** в сервисе-источнике: запись в БД + событие в outbox в одной транзакции; отдельный процесс публикует в брокер.

### 3.4 Read-модели

Для витрины и отчётов допустимы **проекции** (CQRS-lite): материализованные представления в PostgreSQL или Elasticsearch/OpenSearch для полнотекста по каталогу.

### 3.5 Текущий проект

Сейчас много доменов сидит в одной PostgreSQL-схеме gateway — для рефакторинга естественный путь: **сначала модули с чёткими пакетами и границами**, потом вынос с отдельной БД и синхронизацией событий.

## 4. Детальная система кастомных заказов

Цель: учитывать **все компоненты**, **заказы**, **состояния согласования** и связь со **складом** и **производством**.

### 4.1 Сущности (концептуально)

- **CustomOrder** (или расширение OMS-заказа флагом `type=custom`):
  - ссылка на клиента, дедлайн, приоритет, ответственный мастер/менеджер.
- **ConfigurationSnapshot** — **неизменяемый** снимок конфигурации на момент оплаты/фиксации (JSON + нормализованные строки для отчётов).
- **ConfigurationLine** — одна строка спецификации: тип узла (case, pcb, switches, keycaps, cable, plate, foam, mods…), ссылка на SKU или custom line, количество, цена, заметки.
- **CompatibilityRule** — правила из каталога (например, «данный case принимает только plate X или Y»).
- **BOMExplosion** — развёртка в **потребность в компонентах** для склада и производства.
- **Reservation** — связь строк заказа с резервами Inventory.
- **WorkOrder** (производство) — связь с CustomOrder, этапы из вашего `ProductionStage`, чек-листы.
- **CustomerApproval** — этапы «клиент подтвердил раскладку / цвет / фото прототипа» с timestamp и версией.

### 4.2 Жизненный цикл (пример)

1. **Draft** — конфигуратор, цена предварительная.
2. **Quoted** — менеджер/система зафиксировала цену и срок (опционально).
3. **AwaitingPayment** — счёт / оплата онлайн.
4. **Paid** → событие → резерв компонентов.
5. **Procurement** — не хватает позиций, созданы заявки поставщику (отдельный подпроцесс).
6. **InBuild** — MES: этапы assembly, testing, lub…
7. **QC** — как у вас `QualityStatus`.
8. **ReadyToShip** / **Shipped** / **Completed**; **Cancelled** с политикой возврата резервов.

Каждый переход — **запись в аудит** (кто/когда/почему), для споров и аналитики.

### 4.3 Учёт «всех компонентов»

- **Уровень SKU**: переключатель как товар со свойствами (бренд, вес, количество в наборе).
- **Уровень модификации**: «луб 45g» как **услуга** с привязкой к трудозатратам и расходникам (смазка) — отдельные строки BOM.
- **Комплекты**: «набор стабов 2u+7u» как один SKU или BOM из отдельных стабов — политика должна быть одна на весь каталог.

### 4.4 Конфигуратор на фронте

- Пошаговый мастер с **живой фильтрацией** по совместимости (запросы к Catalog + кэш).
- Сохранение черновика в аккаунте (и/или по ссылке «поделиться конфигурацией»).
- Отображение **веса**, **ориентировочной даты** на основе загрузки цеха и наличия (агрегация из Inventory + Production).

## 5. Закупки, поставки, отслеживание, контрагенты и связь с сайтом продаж

### 5.1 Закупки и поставки (входящий поток)

- **Потребность** — из складских минимумов, из производственного плана или из нехватки по BOM кастома (см. этап Procurement в разделе 4).
- **Заявка на закупку (PR)** — внутреннее согласование (опционально).
- **Заказ поставщику (PO)** — контрагент-поставщик, строки (SKU/номенклатура поставщика, количество, цена, срок), статусы: черновик → отправлен поставщику → подтверждён → частично/полностью закрыт.
- **Ожидаемая поставка (ASN / inbound shipment)** — одна или несколько партий в пути от поставщика или от хаба; привязка к PO.
- **Приход на склад (GRN)** — фактическое поступление: количество, брак, партии; событие для Inventory (увеличение остатков). Связь PO → приход — основа для **трёхстороннего сверения** (заказ — накладная — счёт), если нужно бухгалтерски.

В репозитории уже есть задел: сущность **поставщика** (`suppliers`); дальше логично нарастить **PO, строки PO, входящие отгрузки и приёмку** без дублирования справочника номенклатуры с `items`.

### 5.2 Отслеживание поставок (трекинг входящих)

- **Учёт на уровне поставки**: номер отслеживания (трек), перевозчик, статусы (создана → в пути → таможня → на складе поставщика → у нас), плановая дата прибытия (ETA), фактическая дата.
- **Источники статусов**: ручной ввод, импорт из API перевозчика/агрегатора, email-парсинг (хуже), CSV — в зависимости от рынка.
- **Уведомления**: Procurement/Notifications оповещает закупки и склад о задержках или прибытии «сегодня».

Отличать от **исходящей** доставки покупателю (OMS/курьер) — это разные сущности и часто разные интеграции.

### 5.3 Контрагенты

- **Единый справочник юридических лиц** (и/или ИП): название, ИНН/аналог, юр. адрес, банковские реквизиты, контакты.
- **Роли**: поставщик, покупатель (B2B), перевозчик, прочее; одна организация может иметь несколько ролей.
- **Связь с операционными сущностями**: PO ссылается на контрагента-поставщика; исходящий счёт B2B — на контрагента-клиента; договор — сроки оплаты, валюта, Incoterms для международных поставок.
- Текущая модель `suppliers` может эволюционировать в **контрагента с ролью Supplier** или остаться представлением «поставщик» поверх общего справочника.

### 5.4 Сайт для продаж и закупки

| Канал | Что показывает / делает |
|-------|-------------------------|
| **Сайт продаж (публичный)** | Каталог, кастом, оплата, ЛК, **статусы заказов и доставки покупателю** — см. раздел 6 ниже. |
| **Операционный UI** | Закупки, список PO, входящие поставки, треки, приёмка, контрагенты, договоры. |

Закупки **не обязаны** быть видны конечному розничному покупателю; если нужна прозрачность «ждём компонент с завода» — это продуктовое решение и отдельные read-модели/флаги в OMS для ЛК.

## 6. Фронтенд для продаж (сайт)

- **Публичный сайт**: SEO, блог, каталог готовых клавиатур, конфигуратор кастома, личный кабинет заказов, оплата, при необходимости трек **исходящей** посылки к клиенту (интеграция с службой доставки).
- **Админка / цех**: другой BFF или те же API с ролями — WMS, производство, **закупки и приёмка**, поддержка.

Технологически текущий **Flutter** может остаться для мобильного приложения или внутренних экранов; для SEO-витрины часто выбирают **web-first** стек (например, Next.js). Это решение продуктовое, не только техническое.

## 7. Наблюдаемость и безопасность

Детальный состав Prometheus/Grafana/экспортеров и связка с Traefik — в **разделе 0**. Общие **enterprise-требования** (IAM, аудит, compliance, HA/DR) — в разделе **Enterprise** в начале документа.

- **Корреляционный ID** и при необходимости **trace ID** (OpenTelemetry) сквозь gateway → сервисы.
- Метрики и логи — как в разделе 0; **алерты по SLO** и по интеграциям (платежи, доставка).
- **Секреты** платежей и SMTP — только в vault/secret manager.
- **Rate limiting** и защита от злоупотреблений на публичных эндпоинтах конфигуратора и оплаты.

## 8. Связь с текущим кодом

- Модели **склада** (`ItemType`, `ItemCategory`) хорошо ложатся на PIM+Inventory после разделения.
- **Поставщики** (`suppliers`) — задел под SRM; расширение: PO, входящие поставки, привязка к приходам.
- **Production** (`Order`, этапы, качество) — ядро MES; для кастома понадобится явная связь «клиентский заказ → производственный заказ → BOM».
- **`users` в gateway** — на целевой архитектуре учётные данные и роли переезжают в **Auth DB**; gateway остаётся **BFF + edge**: TLS, rate limit, агрегация, **проксирование с Bearer**, проверка JWT по **JWKS Auth** (или делегирование в middleware), без дублирования логики паролей.

## 9. Поэтапная миграция (strangler), без «большого взрыва»

Порядок ниже — про **снижение риска и зависимостей**, а не про «урезанный продукт». Каждый этап должен быть **полноценным для своего контура**: контракты, наблюдаемость, откат.

1. **Зафиксировать контракт событий** для заказа и закупок (`OrderCreated`, `Paid`, `Reserved`, `GoodsReceived`, …) и схему **outbox** там, где уже есть транзакционная БД — в том числе до физического выноса сервисов.
2. **Сервис Auth** — отдельный деплой и **Auth DB**: OAuth2/OIDC, роли и scopes, JWKS, клиенты фронтов; миграция учётных записей из текущего `users`; доменные сервисы переводятся на **проверку access token** (локально по ключам или интроспекция); gateway — без хранения паролей.
3. **Изолировать домен Custom/BOM** в пакетах с явным API к складу и производству — без протекания SQL между контекстами.
4. **SRM как полный цикл**: потребность → PO → ожидаемая поставка / ASN → приход (GRN) → движения в Inventory; **отслеживание входящих** (трек, ETA, статусы) — в том объёме, который зафиксирован в требованиях, а не «потом доделаем».
5. **Выделить первым доменным микросервисом** (после Auth) тот bounded context, который даёт максимум отдачи по нагрузке, релизам или командной автономии — чаще **Catalog** или **Inventory**.
6. **Брокер сообщений** и идемпотентные консьюмеры на критичных цепочках (оплата → резерв → производство/отгрузка).
7. **Витрина и платежи** — интеграция уровня **продакшена**: webhooks, ретраи, идемпотентность, метрики и алерты по провайдеру, не демо-сценарий.
