# Требования к фронтендам (Market и System)

Документ задаёт **обязательную планку качества UI/UX** для двух веб-клиентов на Flutter: витрина (`frontend/market`) и внутренняя система (`frontend/system`). Оба — **только web**, единая дизайн-система с допустимыми отличиями по сценариям.

---

## 1. Визуальный язык и платформа

1. **Material 3 (Material You)** — единственная целевая дизайн-система: компоненты, токены, типографика и поведение из спецификации M3. Кастомные виджеты допускаются только там, где стандартного паттерна M3 недостаточно, с сохранением тех же принципов (роли цвета, скругления, состояния).
2. **Современный интерфейс** означает: предсказуемая иерархия, читаемая сетка, осмысленные отступы, согласованные скругления и elevation/«подъём» поверхностей по ролям M3, анимации короткие и по делу (не «ради красоты»).
3. **Адаптивность**: корректная работа от мобильной ширины в браузере до широких мониторов (breakpoints, перестроение навигации, таблицы/списки без горизонтального хаоса).

### 1.1 Аутентификация (витрина vs операционка)

1. **Market (витрина):** пользователь должен иметь возможность войти **через Google (Gmail)** и **через Telegram** — кнопки, редиректы и состояния загрузки/ошибок в духе M3; контракт с бэкендом — `auth-service` и отдельный OAuth client для витрины (см. `docs/microservices/01-auth.md`, §2).
2. **System (операционка):** вход **только** по **выданной организацией** учётной записи (логин/пароль; при внедрении — корпоративный SSO). **Не** предлагать вход персонала через Telegram/Google «как на сайте»; UI логина отделён от витрины по смыслу и брендингу.

---

## 2. Тема для OLED: тёмная тема с фоном `#000000`

1. **Фон приложения (scaffold / root)** — строго **`#000000`**. Это базовый «чёрный экран» OLED; от него отталкиваются все остальные поверхности.
2. **Не заливать весь UI одним чёрным**: карточки, панели, модальные окна, поля ввода и навигация должны использовать **роли Material 3** (`surface`, `surfaceContainer*`, `primaryContainer` и т.д.) так, чтобы:
   - контент визуально отделялся от фона;
   - сохранялась глубина и читаемость без «плоской серой каши».
3. **Подъём поверхностей** на OLED: предпочтительно выражать через **слегка более светлые оттенки** surface-контейнеров и чёткие границы/outline при необходимости, а не грубые тени — на чистом чёрном тени почти не видны.
4. **Светлая тема** не является целевой для первой версии; если позже понадобится — отдельное решение и отдельная спека (не смешивать с OLED-требованиями).

---

## 3. Цвет и контраст

1. **Контраст текста и иконок** — не ниже разумного минимума для веб: основной текст и интерактивные элементы на выбранных surface должны оставаться читаемыми; проверка **WCAG 2.1** для критичных пар «текст / фон» (желательно уровень AA там, где это реально на тёмных surface рядом с `#000000`).
2. **Акцент (primary / secondary)** — согласован с M3, не «кислотный» без причины; состояния (hover, focus, pressed, disabled) должны быть различимы.
3. **Семантика**: успех, предупреждение, ошибка, информация — отдельные цветовые роли, не только иконка.

---

## 4. Типографика и плотность

1. **Типографика M3**: масштаб `textTheme` согласован с макетной иерархией (display / headline / title / body / label).
2. **Межстрочные интервалы и длина строки** — удобочитаемость на больших экранах (ограничение ширины основного текстового потока где уместно).
3. **Плотность интерфейса**: для **System** допускается более компактная табличная/админская плотность при сохранении кликабельных зон и доступности; **Market** — чуть более «воздушный» retail-UX, без потери структуры.

---

## 5. Компоненты и паттерны UX

1. **Навигация**: явная, с понятным местоположением пользователя (breadcrumb или аналог для глубоких сценариев в System; для Market — понятные разделы витрины).
2. **Формы**: лейблы, подсказки, ошибки валидации у полей; фокус и клавиатура (Tab-порядок) для web.
3. **Состояния экранов**: загрузка, пусто, ошибка сети, «нет прав» — отдельные осмысленные UI, не пустой экран.
4. **Обратная связь**: кнопки и интерактив реагируют на наведение/нажатие; длительные операции — индикатор прогресса или skeleton по правилам M3.
5. **Тактильность**: размеры целей нажатия не меньше разумного минимума для web; не перегружать одним экраном.

---

## 6. Доступность и веб-специфика

1. **Семантика и клавиатура**: фокус видимый; интерактивные элементы — через корректные роли; где нужно — `Semantics`.
2. **Масштаб**: адекватное поведение при увеличении масштаба браузера (без «ломания» критичных layout’ов).
3. **Производительность web**: разумный размер бандла, отсутствие лишних тяжёлых анимаций на слабых машинах; картинки с разумным весом и lazy-loading где уместно.

---

## 7. Согласованность Market vs System

1. **Общее**: одна OLED-тёмная база (`#000000`), Material 3, общие принципы отступов, типографики и компонентного поведения.
2. **Допустимые отличия**:
   - **Market** — акцент на витрине, каталоге, доверии к покупке;
   - **System** — акцент на данных, таблицах, фильтрах, скорости оператора.
3. **Бренд**: допускается различие акцентного seed/брендового цвета между приложениями, но в рамках одной семьи токенов (не два разных «мира»).

---

## 8. Реализация во Flutter

1. Тема оформляется через **`ThemeData` / `ColorScheme` Material 3** (`useMaterial3: true`), с явным заданием scaffold/background **`Color(0xFF000000)`** и настройкой остальных ролей под контраст и OLED.
2. Не хардкодить разрозненные цвета по виджетам без необходимости: опираться на тему и `ThemeExtension` при появлении проектных токенов.
3. Документировать в README фронта кратко: «OLED dark, M3, см. `docs/frontend-requirements.md` в корневом репозитории».

---

## 9. Критерии приёмки (чеклист)

- [ ] Включён Material 3, визуально согласованный с гайдлайнами M3.
- [ ] Корневой фон приложения — `#000000`, контент читаемый, поверхности различимы.
- [ ] Есть продуманные состояния: loading / empty / error / no permission.
- [ ] Адаптивность и клавиатурная навигация на web не ломают сценарии.
- [ ] Market и System не выглядят как два случайных продукта — общая дизайн-логика соблюдена.
