# Git workflow: монорепозиторий и субмодули

Цель — **предсказуемые ветки**, **issue → branch → PR → merge в `dev`**, релизы из `dev` в `master`/`main`.

## Корневой репозиторий `ZhuchkaKeyboards`

| Ветка | Назначение |
|-------|------------|
| `main` | Стабильная линия, теги релизов (по политике). |
| `dev` | Интеграция документации, ссылок на субмодули, compose, инфраструктура репозитория. |
| `feature/<issue>-<slug>` | Короткоживущие ветки под issue (доки, скрипты, `.gitmodules`). |

**Правило:** изменения в корне сначала в **issue**, затем ветка от `dev`, **PR в `dev`**, ревью, merge.

## Каждый субмодуль (сервис или фронт)

| Ветка | Назначение |
|-------|------------|
| `master` | Релизная / стабильная линия сервиса. |
| `dev` | Ежедневная интеграция фич; **все PR сюда** до релиза. |
| `feature/<issue>-<slug>` | Одна фича / одна задача из issue. |
| `fix/<issue>-<slug>` | Багфиксы. |

**Именование веток:** латиница, дефисы, префикс `feature/` или `fix/`, по возможности номер issue: `feature/42-openapi-v1`.

**Pull request:**

1. Ветка от актуального `dev`.
2. PR **в `dev`** (не в `master`).
3. В описании PR: `Closes #N` или `Refs #N`.
4. После merge — ветку фичи удалить (локально и на origin при необходимости).

**Релиз сервиса:** по договорённости merge `dev` → `master` (или tag с `master`), CI/deploy с `master`.

### `services/auth` и `bots/auth_bot`

Для этих двух субмодулей действует **жёсткое правило на уровне issue:**

| Шаг | Что делаем |
|-----|------------|
| 1 | Одна **issue** в трекере — одна **ветка** в соответствующем репозитории (`feature/<N>-<slug>` или `fix/<N>-<slug>`). Не смешивать несколько несвязанных задач в одной ветке. |
| 2 | **Тесты** (pytest и т.п.) добавляются или обновляются **в рамках той же issue** — покрывают именно этот изменённый код, а не «свалку» из разных тикетов. |
| 3 | Один **pull request** закрывает одну логическую задачу (и одну ветку). PR в **`dev`** субмодуля, в описании `Closes #N` / `Refs #N`. |
| 4 | После merge — **bump указателя** субмодуля в корне монорепо отдельным коммитом (как в разделе «Команды»). |

Если затронуты **и** auth, **и** бот (контракт API, конфиг) — заводят **две issue** (или связанные подзадачи) и **два PR** в двух репозиториях, либо одна эпик-issue со ссылками, но ветки и PR всё равно раздельные по репо.

## Issues

- Один issue — одна **закрываемая** единица работы (этап, фича, баг).
- Метки по желанию: `enhancement`, `documentation`, `infra`, `blocked`.
- Крупные эпики — отдельный issue-«контейнер» со списком ссылок на подзадачи.

## Команды (напоминание)

```bash
# Обновить все субмодули
git submodule update --init --recursive
git pull
git submodule update --remote --merge   # осторожно: тянет последние коммиты

# В субмодуле перед работой
git checkout dev && git pull origin dev
git checkout -b feature/12-add-health-metrics

# После merge PR — обновить указатель в монорепозитории
cd ../..
git add services/<name>
git commit -m "chore(submodule): bump <name> after #12"
```

## Чеклист: bump субмодуля после релиза сервиса

Используйте при обновлении указателя в корне после релиза или значимого merge в субмодуле.

1. В репозитории сервиса: нужный коммит в **`master`** (или тег релиза), пайплайн на этой линии **зелёный**.
2. В монорепозитории: обновить субмодуль до целевого SHA — `git submodule update --remote services/<name>` (осторожно с веткой по умолчанию) или `cd services/<name>`, `git fetch`, `git checkout <tag|SHA>`, затем в корне зафиксировать путь субмодуля.
3. Проверить **`git diff services/<name>`** — только ожидаемые изменения.
4. В корне монорепозитория — **отдельный коммит** только с указателем, например: `chore(submodule): bump <name> after release …` или `… after #N`.
5. **Pull request в `dev`** с описанием и ссылкой на PR/релиз/тег в репозитории сервиса.
6. При необходимости обновить **сопутствующую документацию** в монорепозитории и снимки OpenAPI (см. [openapi-sync.md](openapi-sync.md)).

## Скрипты в репозитории

- `scripts/sync-openapi-snapshot.ps1` — сохраняет `GET /api/openapi.json` с запущенного сервиса в `docs/openapi/snapshots/` (см. [openapi-sync.md](openapi-sync.md)).
- `scripts/bootstrap-dev-branches.ps1` — создаёт/обновляет ветку `dev` во всех субмодулях от `master`.
- `scripts/gh-create-issues.ps1` — создаёт стартовый набор issues через GitHub CLI (`gh`), **идемпотентность не гарантируется** (повторный запуск создаст дубликаты). Запускать один раз после установки `gh auth login`.
- `scripts/git-rewrite-history.sh` (Git Bash) — переписывает историю: убирает строку `Made-with: Cursor` из сообщений и приводит автора к `git config user.name` / `user.email` для совпадающих email (остальные авторы не трогает). Запуск из корня монорепо или с путём к клону: `bash scripts/git-rewrite-history.sh services/auth`. После использования нужен **`git push --force-with-lease`** на затронутые ветки; делать только осознанно.
