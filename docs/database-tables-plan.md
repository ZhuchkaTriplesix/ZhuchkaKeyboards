# План таблиц баз данных (целевой enterprise-контур)

Документ задаёт **логический состав таблиц** по bounded context’ам. В продакшене — **отдельные БД или схемы** на сервис (см. `architecture-and-domain-thoughts.md`); здесь имена таблиц и связи описаны без привязки к одной физической БД.

**Условные обозначения**

- **[есть]** — таблица уже заведена в миграции `gateway/.../0001_initial_schema.py`.
- **[новая]** — к добавлению в целевой модели (имена можно уточнить при проектировании).

---

## 1. Identity / IAM / аудит — **БД сервиса Auth**

Таблицы этого блока в продакшене живут в **отдельной базе (Auth DB)** у **сервиса аутентификации и авторизации**, а не в gateway и не в доменных сервисах. Сейчас `users` физически в миграции gateway — при выносе Auth переносится вместе с данными.

| Таблица | Назначение |
|---------|------------|
| `users` | **[есть]** Учётные записи (логин, хеш пароля, контакты). |
| `user_identities` | **[новая]** Внешние провайдеры (OIDC `sub`, привязка к `users`). |
| `organizations` | **[новая]** Юрлица/филиалы компании (владелец складов, договоров закупки/продажи). |
| `roles` | **[новая]** Справочник ролей. |
| `permissions` | **[новая]** Детализация прав (ресурс + действие). |
| `role_permissions` | **[новая]** M2M роль ↔ право. |
| `user_roles` | **[новая]** M2M пользователь ↔ роль (+ опционально `organization_id`, scope). |
| `sessions` / `refresh_tokens` | **[новая]** Сессии / refresh для веб и API-клиентов. |
| `audit_log` | **[новая]** Неизменяемые записи: субъект, объект, действие, до/после, correlation_id, timestamp. |
| `service_accounts` | **[новая]** Технические учётки для интеграций (ротация ключей). |
| `oauth_clients` | **[новая]** Зарегистрированные клиенты OAuth2 (фронты, mobile, confidential server clients). |
| `oauth_authorization_codes` / `oauth_tokens` | **[новая]** Коды авторизации, refresh-токены (если не делегированы полностью внешнему IdP). |

## 2. Контрагенты (MDM) и договоры

| Таблица | Назначение |
|---------|------------|
| `legal_entities` | **[новая]** Контрагент: ИНН/аналог, юр. имя, адрес, страна, реквизиты. |
| `legal_entity_roles` | **[новая]** Роли: supplier, customer_b2b, carrier, internal — M2M с `legal_entities`. |
| `suppliers` | **[есть]** Расширить FK на `legal_entities` (или слить с ними по мере миграции). |
| `contracts` | **[новая]** Договор с контрагентом: номер, срок, валюта, условия оплаты, Incoterms. |
| `contract_lines` | **[новая]** Опционально: позиции рамочного договора (SKU/категории). |
| `addresses` | **[новая]** Юр/почтовые адреса, привязка к контрагенту и типу. |
| `contacts` | **[новая]** Лица контакта у контрагента (для закупок и B2B). |

## 3. Каталог / PIM (витрина)

Отдельно от складских `items`: то, что видит покупатель (SEO, медиа, совместимость).

| Таблица | Назначение |
|---------|------------|
| `products` | **[новая]** Торговая сущность (название, slug, описание, статус публикации). |
| `product_variants` | **[новая]** Варианты (цвет, комплектация), связь с `items.sku` или отдельным SKU. |
| `categories` | **[новая]** Древовидная или плоская категоризация. |
| `product_categories` | **[новая]** M2M продукт ↔ категория. |
| `product_media` | **[новая]** URL, порядок, alt — или ссылка на объект в object storage + метаданные. |
| `attributes` / `attribute_values` | **[новая]** EAV или JSON-атрибуты для фильтров (layout, размер case и т.д.). |
| `compatibility_rules` | **[новая]** Правила совместимости (variant A ↔ variant B); для конфигуратора. |

## 5. OMS — заказы продажи

| Таблица | Назначение |
|---------|------------|
| `customers` | **[новая]** Покупатель B2C/B2B: FK на `users`, опционально `legal_entity_id`. |
| `sales_orders` | **[новая]** Заказ продажи: номер, статус, доставка, оплата, суммы, валюта, канал. |
| `sales_order_lines` | **[новая]** Строки: ссылка на variant/конфигурацию, qty, цена, налоги. |
| `sales_order_status_history` | **[новая]** История статусов для SLA и ЛК. |
| `shipments` | **[новая]** Исходящие отгрузки покупателю: перевозчик, трек, ETA. |
| `shipment_lines` | **[новая]** Состав отгрузки по строкам заказа. |
| `shipment_tracking_events` | **[новая]** События трекинга (внешний API / ручной ввод). |

Связь с производством: `sales_orders` → `work_orders` (см. раздел 8).

## 13. Сводка: что уже есть vs объём доработки

**Уже в одной миграции (ядро склада + производство + базовые закупки):**  
`users`, `warehouses`, `warehouse_zones`, `suppliers`, `items`, `supplier_items`, `purchase_orders`, `purchase_order_items`, `orders`, `production_tasks`, `resources`, `task_resources`, `quality_checks`, `quality_checklist_items`, `inventory_levels`, `inventory_transactions`.

**Крупные блоки новых таблиц:** IAM и аудит, контрагенты и договоры, PIM, commerce/OMS, платежи, полный контур входящих поставок (ASN, GRN, трекинг), явные резервы, кастом/BOM, уведомления, outbox/идемпотентность.

## 14. Индексы и качество данных (на этапе физического проектирования)

- Внешние ключи на всех связях «заказ → строка → резерв/движение».
- Уникальные ограничения: номера заказов, PO, публичные slug продуктов.
- Частичные индексы под активные корзины, необработанный outbox, pending payments.

*Документ — план для согласования доменов; физическая нарезка по БД — по границам микросервисов.*
