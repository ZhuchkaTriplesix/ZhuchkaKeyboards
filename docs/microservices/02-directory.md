# Directory / Customer service (`services/directory`) — детальные требования

## 1. Назначение и границы

- Владелец **профиля клиента** после аутентификации: связь с `sub` из **Auth**, B2C/B2B атрибуты, **адреса**, **согласия** (ПДн, маркетинг), предпочтения.
- **Не владелец**: учётных записей и паролей; юридического лица как справочника (это `counterparties`) — только **ссылки**.

## 2. Владение данными

| Сущность | Описание |
|----------|----------|
| `Customer` | `subject` (FK логически на Auth), тип B2C/B2B, display_name, телефон, locale, timezone |
| `CustomerAddress` | Тип (billing/shipping), строки адреса, страна, индекс, по умолчанию |
| `ConsentRecord` | Тип согласия (privacy, marketing, …), версия текста, `granted_at`, `withdrawn_at`, источник |
| `B2BLink` | `counterparty_id` + роль контакта при B2B |

Идентификатор клиента в API — **`customer_id`** (UUID); маппинг `subject` ↔ `customer_id` хранится только здесь.

## 3. API — публичный контур (владелец токена)

Базовый префикс: `/api/v1`.

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/me` | Текущий профиль по Bearer (resolve по `sub`) |
| PATCH | `/me` | Обновление полей разрешённых самим пользователем |
| GET | `/me/addresses` | Список адресов |
| POST | `/me/addresses` | Создать адрес |
| PATCH | `/me/addresses/{id}` | Изменить |
| DELETE | `/me/addresses/{id}` | Удалить (если не используется в активном заказе — доменное правило) |
| GET | `/me/consents` | Текущие согласия |
| POST | `/me/consents` | Выдать/обновить согласие с версией документа |

**Scopes:** минимум `profile` или `customer`; без токена — только редирект на Auth.

## 4. API — операционный контур (персонал)

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/customers` | Список с фильтрами (email, subject, counterparty_id) |
| GET | `/customers/{id}` | Карточка |
| PATCH | `/customers/{id}` | Корректировки (с аудитом) |
| POST | `/customers/{id}/merge` | Слияние дублей (редко; идемпотентная операция) |

Роли: `support.read`, `support.write` или аналог.

## 5. Интеграции

- **Auth**: при первом входе — создание `Customer` по событию `auth.user.created` или lazy при первом вызове `/me` (политика должна быть одна на платформу).
- **Counterparties**: чтение реквизитов для отображения B2B; запись только ссылок `counterparty_id`.
- **OMS/Commerce**: адрес доставки передаётся **снимком** в заказе; изменение адреса в Directory не меняет уже созданный заказ.

## 6. События

| Событие | Когда |
|---------|--------|
| `directory.customer.created` | Первичное создание профиля |
| `directory.customer.updated` | Изменение данных |
| `directory.consent.changed` | Юридически значимое изменение согласия |

## 7. ПДн и compliance

- Минимизация в ответах API; экспорт/удаление по запросу субъекта — эндпоинты или задание в очередь (уточнить политику).
- Логи без полного телефона/адреса в plain text.

## 8. Критерии приёмки

- [ ] `/me` стабильно маппится на Auth `sub`.
- [ ] Согласия версионируются; история доступна для аудита.
- [ ] OpenAPI с разделением «self» и «staff» операций.
