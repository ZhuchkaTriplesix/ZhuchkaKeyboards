# Notification service (`services/notification`) — детальные требования

## 1. Назначение и границы

- Владелец **доставки сообщений** в каналы: **email**, **SMS**, **push** (если появятся мобильные), **webhook**, опционально мессенджеры.
- Владелец **шаблонов**, **очереди**, **статусов** доставки и **идемпотентности** на уровне «не отправить одно и то же дважды».
- **Не владелец**: бизнес-логикой заказа; только подписывается на события или принимает явные запросы на отправку.

## 2. Владение данными

| Сущность | Описание |
|----------|----------|
| `Template` | Ключ, канал, версия тела, локали |
| `NotificationRequest` | Получатель, payload, канал, `idempotency_key` |
| `DeliveryStatus` | queued / sent / failed / bounced |
| `SuppressionList` | Отписка email/SMS (compliance) |

## 3. API

Префикс `/api/v1`.

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/notifications` | Явная отправка: `template_key`, `to`, `variables`, `idempotency_key` **обязателен** для массовых триггеров |
| GET | `/notifications/{id}` | Статус |
| CRUD | `/templates` | Управление (staff) |
| POST | `/suppressions` | Добавить отписку |

### Внутренний consumer

- Worker читает брокер: `oms.order.shipped` → шаблон `order_shipped` и т.д. Маппинг **конфигурируемый** (таблица routing rules).

## 4. Каналы — требования

- **Email**: SPF/DKIM/DMARC на стороне домена; bounce handling → suppression.
- **SMS**: лимиты провайдера; не отправлять на запрещённые регионы без политики.
- **Webhook**: подпись HMAC, retry с backoff, DLQ.

## 5. События

### Входящие (подписка)

- Из OMS, Payments, Fulfillment, Auth (опционально), Custom.

### Исходящие (опционально)

- `notification.delivered`, `notification.failed` — для аналитики.

## 6. Нефункциональные требования

- Очередь устойчива к пикам (чёрная пятница).
- PII в payload — минимизация; TTL хранения содержимого писем по политике.

## 7. Критерии приёмки

- [ ] Один и тот же `idempotency_key` не создаёт второе письмо.
- [ ] Шаблоны версионируются; при смене версии старые рассылки не ломаются (pin версии в истории).
- [ ] Ошибки провайдера не теряются: retry и финальный failed статус.
