# Inventory service (`services/inventory`) — детальные требования

## 1. Назначение и границы

- **Источник истины по количествам**: остатки по **SKU** и **локации** (или складу агрегированно — уточнить модель), **резервы**, **движения** (приход/списание/корректировка).
- **Не владелец**: ячейками и задачами отбора (WMS), жизненным циклом заказа (OMS) — только реагирует на команды/события.

## 2. Владение данными

| Сущность | Описание |
|----------|----------|
| `StockBalance` | sku_id, location_id?, qty_on_hand, qty_reserved |
| `Reservation` | order_id или link, sku_id, qty, статус (active/released/consumed) |
| `StockMovement` | Тип, qty, reason, ref (order, po, adjustment), timestamp, actor |
| `Adjustment` | Инвентаризационная корректировка с approval |

Инварианты: `qty_on_hand >= 0` (если политика не допускает минус-бэкордер на уровне inventory).

## 3. API

Префикс `/api/v1`.

### Чтение

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/availability` | query: `sku_ids[]`, `location_id?` → доступно к продаже |
| GET | `/stock` | Остатки с фильтрами (операционка) |
| GET | `/movements` | Журнал движений |

### Запись (строго RBAC / service account)

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/reservations` | Зарезервировать под заказ; **Idempotency-Key** обязателен |
| POST | `/reservations/{id}/release` | Снять резерв |
| POST | `/reservations/{id}/commit` | Списать в отгрузку |
| POST | `/adjustments` | Корректировка с причиной |
| POST | `/receipts` | Приход от закупки/производства (связь с документом источника) |

Каждая операция, меняющая количество, порождает **неизменяемую** запись в `StockMovement`.

## 4. События

### Исходящие

| Событие | Когда |
|---------|--------|
| `inventory.reserved` | Успешный резерв |
| `inventory.released` | Снятие |
| `inventory.stock_low` | Ниже порога |
| `inventory.movement.posted` | Любое движение (для аналитики) |

### Входящие

- `oms.order.paid` → создание резерва (если политика резерва на оплату).
- `procurement.goods.received` → приход количества.

## 5. Идемпотентность и гонки

- Резерв с тем же ключом и теми же параметрами возвращает тот же `reservation_id`.
- Оптимистичная блокировка строки баланса или serializable транзакция при изменении одного SKU в одной локации.

## 6. Критерии приёмки

- [ ] Нет «пропавших» единиц: сумма движений согласована с балансом.
- [ ] Двойной резерв под один заказ невозможен при корректном idempotency.
- [ ] Операционный журнал движений экспортируется для аудита.
