# Counterparties service (`services/counterparties`) — детальные требования

## 1. Назначение и границы

- Владелец **мастер-данных контрагентов**: юрлица/ИП, **роли** (клиент B2B, поставщик, перевозчик), **договоры**, **реквизиты**, платёжные условия, контакты.
- **Единая точка ссылки** для Procurement, Directory (B2B), Fulfillment (перевозчики).

## 2. Владение данными

| Сущность | Описание |
|----------|----------|
| `Counterparty` | ИНН/идентификатор страны, юр. название, адрес |
| `BankAccount` | Реквизиты; маскирование в API для непривилегированных |
| `Contract` | Номер, период, условия, сканы в object storage |
| `PartyRole` | supplier / b2b_customer / carrier / … |
| `Contact` | ФИО, email, телефон, роль |

## 3. API

Префикс `/api/v1`.

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/counterparties` | Создание с ролями |
| GET | `/counterparties` | Поиск: ИНН, имя, роль |
| GET | `/counterparties/{id}` | Карточка |
| PATCH | `/counterparties/{id}` | Обновление |
| CRUD | `/counterparties/{id}/contracts` | Договоры |
| CRUD | `/counterparties/{id}/bank-accounts` | С ограничением отображения |

### Чтение для других сервисов

- `GET /internal/counterparties/{id}/validate?role=supplier` — лёгкая проверка для Procurement.

## 4. События

| Событие | Когда |
|---------|--------|
| `counterparty.updated` | Любое изменение реквизитов |
| `contract.activated` | Новый договор |

## 5. ПДн и секреты

- Банковские реквизиты — шифрование at rest; в логах не выводить.
- Разграничение: закупки видят полные реквизиты поставщика; витрина — нет.

## 6. Критерии приёмки

- [ ] Дубликат ИНН в одной юрисдикции — запрет или merge workflow.
- [ ] История изменений реквизитов для финансового аудита.
- [ ] OpenAPI с маскированием полей по ролям.
