# WMS service (`services/wms`) — детальные требования

## 1. Назначение и границы

- Владелец **складской топологии**: склады, зоны, **ячейки (bins)**, правила размещения.
- **Операции**: приёмка (inbound), отбор (picking), размещение (putaway), перемещения, **циклическая инвентаризация**.
- **Не владелец**: итоговым количеством в учёте — подтверждение движений инициирует изменения в **Inventory** (одна транзакция или saga).

## 2. Владение данными

| Сущность | Описание |
|----------|----------|
| `Warehouse` | Код, адрес, часовой пояс |
| `Zone` | Тип (storage, staging, quarantine) |
| `Bin` | Координата (aisle-rack-level), ёмкость, ограничения |
| `InboundShipment` | Ожидаемое поступление (от Procurement), строки |
| `Task` | Тип pick/putaway/count, assignee, статус |
| `CycleCount` | Сессия инвентаризации по зоне |

## 3. API

Префикс `/api/v1`.

### Справочники

| Метод | Путь | Описание |
|-------|------|----------|
| CRUD | `/warehouses`, `/zones`, `/bins` | Управление топологией |

### Операции

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/inbound` | Зарегистрировать ожидаемую поставку |
| POST | `/inbound/{id}/receive` | Подтвердить приём по строкам (скан SKU/qty) |
| POST | `/tasks/pick` | Создать задание отбора под shipment/волну |
| PATCH | `/tasks/{id}/complete` | Завершить с фактом |
| POST | `/cycle-counts` | Запуск пересчёта |

## 4. Связь с Inventory

- При **receive**: после валидации — вызов `inventory.receipts` или событие `wms.goods.received` → Inventory увеличивает остаток.
- При **pick complete**: `inventory.commit` резерва или списание по правилу.
- **Сага или локальная транзакция** только в одном сервисе невозможна — нужен **outbox** и упорядоченная обработка.

## 5. События

| Событие | Когда |
|---------|--------|
| `wms.task.completed` | Для триггера Fulfillment |
| `wms.inbound.received` | Приход на склад |
| `wms.cycle_count.variance` | Расхождение → корректировка Inventory |

## 6. UX оператора (косвенные требования к API)

- Поддержка **сканера штрихкода**: ввод `bin_code`, `sku` в эндпоинтах confirm.
- Идемпотентность на `receive`/`complete` по `scan_session_id` + line.

## 7. Критерии приёмки

- [ ] Каждая ячейка уникальна в рамках склада.
- [ ] Повтор `receive` не удваивает приход.
- [ ] Задачи имеют аудит (кто, когда закрыл).
