# Payments service (`services/payments`) — детальные требования

## 1. Назначение и границы

- Единственная точка интеграции с **PSP** (Stripe, ЮKassa, …): **PaymentIntent**, capture, **refund**, webhooks.
- **PCI**: PAN/CVC **не** проходят через другие сервисы; токены провайдера только здесь.
- **Не владелец**: бизнес-статусом заказа — только **платёжным** состоянием и событиями для OMS.

## 2. Владение данными

| Сущность | Описание |
|----------|----------|
| `PaymentIntent` | Внешний id провайдера, сумма, валюта, статус, `order_id`, `idempotency_key` |
| `Refund` | Связь с intent, сумма, статус |
| `WebhookEvent` | Сырые payload с подписью, обработка идемпотентная по `event_id` провайдера |

Хранить только **минимум** данных по PCI DSS (см. документацию PSP).

## 3. API — исходящий контракт для фронта/BFF

Префикс `/api/v1`.

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/payment-intents` | Создать намерение: `order_id`, amount, currency, `Idempotency-Key` **обязателен** |
| GET | `/payment-intents/{id}` | Статус для polling |
| POST | `/payment-intents/{id}/confirm` | Если требуется серверный confirm |
| POST | `/refunds` | Возврат по `order_id` или `payment_intent_id` |

Ответы **не** содержат секретных ключей; для клиентского SDK — только **client_secret** или эквивалент, который разрешено отдавать в браузер по политике PSP.

## 4. Webhooks (входящие от PSP)

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/webhooks/{provider}` | Верификация подписи; быстрый 200; обработка асинхронно из очереди при необходимости |

Обработка:

- Идемпотентность по id события провайдера.
- Переходы: `authorized` → `captured` / `failed`; обновление OMS через событие или синхронный вызов (предпочтительно **событие** `payments.payment.captured`).

## 5. События (исходящие)

| Событие | Когда |
|---------|--------|
| `payments.intent.created` | Для аудита |
| `payments.capture.succeeded` | Успешное списание |
| `payments.capture.failed` | Отказ |
| `payments.refund.completed` | Возврат завершён |

OMS подписывается и переводит заказ в оплаченный / частично возвращённый.

## 6. Безопасность

- Секреты webhook — ротация; отдельный секрет на dev/stage/prod.
- Логи без сырых PAN; токены провайдера маскировать.
- Rate limit на создание intent по `order_id` + IP.

## 7. Критерии приёмки

- [ ] Повтор `POST /payment-intents` с тем же `Idempotency-Key` возвращает тот же intent.
- [ ] Webhook обработан ровно один раз на `event_id`.
- [ ] Ни один другой сервис не хранит реквизиты карты.
